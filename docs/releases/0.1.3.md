# Release 0.1.3

## Summary

`ccw-mcp` `0.1.3` introduces lane-aware compilation for ranked files and expands
default multi-harness context detection.

## What changed

- Split ranked file selection into two explicit lanes:
  - `## Files` for task evidence (implementation and test files relevant to the task)
  - `## Agentic Context` for repo/harness instruction and contract files
- Reserved lane budget so agentic files do not crowd out task evidence in the
  main ranked file set.
- Expanded harness/context detection to cover common ecosystems and layouts,
  including `.cursor`, `.codex`, `.claude`, and GitHub instruction paths.
- Improved exclusions for vendored and third-party trees so copied instruction
  files do not dominate rankings.
- Preserved tail-biased snippet extraction for `wiki/user/log.md` so recent
  operational/project history is favored.

## Why this release matters

Before `0.1.3`, instruction-heavy repositories could over-promote agentic files
in the same ranked set used for task evidence. This release makes context
selection more robust for real implementation tasks while still surfacing
repository contracts and harness guidance.

## Client update guide

Update your installation according to how your client launches CCW.

### pip

```bash
pip install -U ccw-mcp
```

### pipx

```bash
pipx upgrade ccw-mcp
```

### uvx launchers

```bash
uvx --refresh ccw-mcp==0.1.3 --help
```

### APM-managed MCP clients

```bash
apm install fmferrari/ccw --target copilot
```

If you target multiple harnesses, run install per target as needed.

## Post-update verification

1. Restart the editor/harness process that hosts the MCP server.
2. Re-index the target repo:

```bash
ccw index .
```

3. Compile a fresh artifact and confirm lane sections exist:

```bash
ccw compile --task "Verify lane behavior" --out .ccw/compiled/lane-check.md
rg "^## " .ccw/compiled/lane-check.md
```

Expected section headers include:

- `## Files`
- `## Agentic Context` (when qualifying context files are present)

## Known migration pitfall

If lane behavior looks unchanged after upgrade, the client is usually still
running an older server process. Refresh package/runtime, then restart the
harness and regenerate context.
